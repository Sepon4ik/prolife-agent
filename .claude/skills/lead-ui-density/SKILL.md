# Lead UI Density — Showing 80+ fields without overwhelming

## When to use

When designing or implementing any UI that displays a **rich entity** with dozens to hundreds of fields — typically a lead/contact/account profile in B2B tools, but the same principles apply to any data-dense view (a customer profile in support, a patient record, an asset in a portfolio).

The pain this skill solves: ProLife Agent (and any similar product) collects huge amounts of data per lead from scraping, enrichment, scoring, outreach history, intent signals, and reply analysis. The naive UI is a vertical wall of fields that nobody can read. The right UI is **layered, scannable, and action-oriented**.

This is a sister skill to `lead-data-fusion` — that one builds the data, this one shows it.

---

## The core insight

Users don't want to see "all the fields." Users want to **make a decision** about a lead in 5 seconds. The UI's job is to:

1. **Tell them whether this lead is worth their time** in one glance
2. **Surface the next obvious action** without scrolling
3. **Let them drill down** when they want more detail
4. **Get out of the way** when they want to move to the next lead

Every UI decision should trace back to one of those four jobs.

---

## The 4-layer architecture

```
Layer 1 — At-a-glance card (1 row in a list)
  └─ Layer 2 — Hero section (top of detail page, 0 scrolling)
       └─ Layer 3 — Tabbed sections (one click to dig in)
            └─ Layer 4 — Raw data + provenance (one more click)
```

### Layer 1: The list row
This is what the user sees in a table or grid of leads. **Only the fields needed to decide "open / skip / next."**

Recommended fields for a lead row:
- **Name + company** (the human/org)
- **Score** with color (visual sort)
- **Status badge** (new / contacted / replied / qualified / dead)
- **Last activity** (relative time: "2 days ago")
- **One key signal** (the most recent intent signal: "viewed pricing 3x", "hired CTO", "raised Series B")
- **Quick actions** on hover (send, snooze, mark dead)

Anti-pattern: a 12-column table with Email, Phone, LinkedIn, Website, City, Country, Title, Company Size, Industry, Tech Stack, Last Email Date, Status. The user can't scan 12 columns. Pick 5.

```tsx
<TableRow>
  <Cell>
    <Avatar src={lead.avatarUrl} fallback={initials(lead)} />
    <div>
      <div className="font-medium">{lead.fullName}</div>
      <div className="text-sm text-muted-foreground">{lead.company.name}</div>
    </div>
  </Cell>
  <Cell><ScoreBadge score={lead.score} /></Cell>
  <Cell><StatusBadge status={lead.status} /></Cell>
  <Cell className="text-sm">{relativeTime(lead.lastActivityAt)}</Cell>
  <Cell><TopSignal signal={lead.topSignal} /></Cell>
  <Cell><QuickActions lead={lead} /></Cell>
</TableRow>
```

### Layer 2: The hero (above the fold on detail page)
The detail page opens. The user has to know **enough to act** without scrolling. Hero = top 600px-800px:

```
┌────────────────────────────────────────────────────────────────────┐
│  [Avatar]  Sarah Chen                                  [⋯ actions]│
│            VP of Marketing at Acme Corp                            │
│            sarah.chen@acme.com  •  linkedin.com/in/sarahchen       │
│                                                                     │
│  ┌─────────┬─────────────┬─────────────┬─────────────────────────┐│
│  │ Score   │ Status      │ Last touch  │ Pipeline stage          ││
│  │  92     │ Contacted   │ 2d ago      │ Awaiting reply          ││
│  └─────────┴─────────────┴─────────────┴─────────────────────────┘│
│                                                                     │
│  💡 Why this lead matters                                          │
│  Strong fit (92/100). Hired a CMO last week. Visited pricing      │
│  page 3x in last 5 days. Replies typically within 24h.             │
│                                                                     │
│  [📧 Send follow-up] [📅 Book meeting] [🔥 Mark hot]  [Snooze ▾]  │
└────────────────────────────────────────────────────────────────────┘
```

Three things in the hero:
1. **Identity row** — who/what/where, the obvious stuff
2. **Status grid** — 4 chips with the most decision-relevant aggregates
3. **Why this matters** — a 1-3 sentence AI-generated summary explaining the score and recent context. **This is the biggest UX win.** Don't make the user read 80 fields to figure out what they're looking at.
4. **Action bar** — 3-4 primary actions, then secondary in a kebab menu

The "why this matters" line is generated by Claude on lead update from the lead's recent events + signals + score factors. Cache it on the Lead row, regenerate on significant changes.

### Layer 3: Tabbed sections
Below the hero, organize the rest into 4-6 tabs. Default to "Overview" or "Activity" — whatever the user opens this page to do most often.

**Recommended tabs for a lead detail view:**

1. **Activity** — chronological timeline of every event (emails sent/opened/clicked/replied, status changes, enrichment runs, manual notes). Default tab — this is what the user comes here to see most often.
2. **Profile** — the long form of identity: full bio, work history, education, social profiles, demographics. Each field shows its source on hover.
3. **Company** — everything about the company. Domain, size, industry, funding, tech stack, news.
4. **Signals** — intent signals over time. Visited pricing page. Hired roles. Press mentions. Tech changes.
5. **Communications** — the actual emails sent and received. Threaded view.
6. **Notes** — manual notes, internal comments, history of who touched the lead.

Avoid going past 6 tabs. If you need 7, you have two products and should split.

### Layer 4: Provenance / raw data
Each field in Layer 2-3 has a hover/click affordance to show its source: where the value came from, when, and what other values existed before.

```
┌─────────────────────────────────────────────┐
│ Job title:  VP of Marketing                 │
│ ─────────────────────────────────────────── │
│ Source: LinkedIn (self-reported)            │
│ Last verified: 2026-03-15                   │
│ Confidence: 0.94                            │
│                                             │
│ Other values seen:                          │
│  • "VP Marketing" (Apollo, 2026-02-01)      │
│  • "Marketing Director" (scraping, 2024-11) │
│                                             │
│ [Edit]  [Lock to manual]                    │
└─────────────────────────────────────────────┘
```

This is the **trust layer**. Power users will check it. Most won't. Both groups need it to exist.

## The "Why this matters" pattern (the most important UI element)

Every dense profile view needs **one short paragraph at the top** that summarizes the lead's state in plain English. Generate it with Claude. Cache it. Regenerate on event.

Prompt template:

```
You are a sales rep's assistant. Write a 2-3 sentence summary of this lead
that helps the rep decide their next action. Be specific. Mention the lead's
fit score and the most relevant recent signals. Do not use marketing language.

Lead: {firstName} {lastName}, {jobTitle} at {companyName} ({companyDomain})
Fit score: {score}/100
Score factors: {topScoreFactors}
Recent events: {last5Events}
Recent signals: {last5Signals}
Status: {status}

Output: 2-3 sentences max. End with a recommended action.
```

Examples of good output:

> Strong fit (92). Sarah leads marketing at a Series B SaaS in your ICP. Visited pricing 3 times in the last week and opened 2 of your last 3 emails without replying. Recommend a soft direct ask.

> Mid fit (61). Pranav is a senior IC at a 10-person agency — slightly below your usual ICP, but they just hired a head of sales which might be a buying signal. Worth a single light-touch outreach, then move on.

> Cold (28). Acme is in your industry but the contact's role doesn't match. Skip unless you find a better contact at the same company.

Good summary saves 30 seconds per lead × 100 leads/day = a lot.

## Density tactics

When a section legitimately needs to show many fields, use these to keep it scannable:

### Tactic 1: Progressive disclosure with "Show more"
Show 6 most important fields. Below that: "Show 23 more fields" expander.

### Tactic 2: Group by semantic category
Don't list 50 fields alphabetically. Group:
- **Identity** (name, title, email, phone, social URLs)
- **Company** (name, domain, size, industry, funding)
- **Location** (city, region, country, timezone)
- **Engagement** (last email, opens, clicks, replies)
- **Tech** (stack, tools, integrations)

A grouped section is scannable. A 50-field flat list is not.

### Tactic 3: 2-column grids for paired data
Paired fields (label + value) work better in 2 columns than as a 1-column form. The eye scans both halves of the row in one fixation.

```tsx
<div className="grid grid-cols-[max-content_1fr] gap-x-4 gap-y-2">
  <dt className="text-muted-foreground">Email</dt>
  <dd>{lead.email}</dd>
  <dt className="text-muted-foreground">Phone</dt>
  <dd>{lead.phone ?? "—"}</dd>
  <dt className="text-muted-foreground">Title</dt>
  <dd>{lead.jobTitle}</dd>
  ...
</div>
```

### Tactic 4: Empty fields disappear
If a field has no value, **don't render it at all** by default. A row that says "Phone: —" is noise. Let "Show empty fields" be an opt-in toggle.

### Tactic 5: Format values, never show raw
- Phones: format with country code (`+1 (415) 555-0100`), not raw `4155550100`
- Dates: relative when recent (`2 days ago`), absolute when old (`Jan 15, 2025`)
- URLs: hostname only with link icon, not `https://www.linkedin.com/in/...?utm_source=...`
- Emails: clickable mailto with copy button on hover
- Numbers: locale-formatted with separators (`1,247` not `1247`)
- Currencies: with symbol and short suffix for big numbers (`$1.2M` not `1200000`)
- Booleans: ✓ / — , not `true` / `null`

### Tactic 6: Inline editing
Make every field editable in place. Click → text input → blur saves. Don't open a modal for one field. Modals are for multi-field forms.

### Tactic 7: Color sparingly, badges for state
Color is a finite resource. Use it for:
- **Status badges** (Hot, Cold, Replied)
- **Score** (red < 40 < orange < 60 < yellow < 80 < green)
- **Errors / warnings**

Don't use color for decoration. Don't use 8 different badge colors. Cap at 5.

## The activity timeline (most-used view, get this right)

The Activity tab is where users live. It must be:

- **Reverse chronological** by default (newest at top)
- **Filterable** (only emails, only signals, only status changes)
- **Compact** — 3-line entries, not 10-line cards
- **Iconographic** — every event type has a glyph so the user can pattern-match types at a glance
- **Grouped by day** with a sticky day header
- **Lazy-loaded** if the timeline is long (most leads have 5-30 events; some have 500)

```tsx
<Timeline>
  <DayGroup date="Today">
    <Event icon={<MailIcon />}>
      You sent <strong>"Following up on your pricing question"</strong>
      <time>2:14 PM</time>
    </Event>
    <Event icon={<EyeIcon />}>
      Sarah opened your previous email
      <time>11:30 AM</time>
    </Event>
  </DayGroup>
  <DayGroup date="Yesterday">
    <Event icon={<SparkleIcon />}>
      Score increased <strong>74 → 92</strong> (visited pricing 3x)
    </Event>
    ...
  </DayGroup>
</Timeline>
```

## Keyboard shortcuts (power users will thank you)

Lead tools live or die on speed. Add keyboard shortcuts:

| Key | Action |
|---|---|
| `j` / `k` | Next / previous lead |
| `e` | Compose email |
| `s` | Snooze |
| `m` | Mark hot/cold |
| `r` | Mark replied |
| `n` | New note |
| `?` | Show shortcut help |
| `/` | Focus search |
| `g a` | Go to Activity tab |
| `g p` | Go to Profile tab |

Implement once. Reuse across all rich-entity views.

## Action bar best practices

The action bar in the hero is the highest-value UI in the entire app. Get it right.

- **3-4 primary actions max.** Pick the most common ones: Send, Book, Mark.
- **Destructive actions are not primary.** Delete, archive, mark-dead → secondary menu.
- **Each action should have keyboard shortcut.**
- **Confirm only when necessary** (delete yes, snooze no).
- **Show busy state** on the button when the action is running, not a global spinner.

```tsx
<ActionBar>
  <Button variant="default" onClick={onSendFollowUp}>
    <MailIcon /> Send follow-up
    <Kbd>e</Kbd>
  </Button>
  <Button variant="outline" onClick={onBookMeeting}>
    <CalendarIcon /> Book meeting
  </Button>
  <Button variant="outline" onClick={onMarkHot}>
    <FireIcon /> Mark hot
    <Kbd>m</Kbd>
  </Button>
  <DropdownMenu>
    <DropdownMenuTrigger asChild>
      <Button variant="ghost">⋯</Button>
    </DropdownMenuTrigger>
    <DropdownMenuContent>
      <DropdownMenuItem onClick={onSnooze}>Snooze</DropdownMenuItem>
      <DropdownMenuItem onClick={onMarkDead}>Mark as dead</DropdownMenuItem>
      <DropdownMenuItem onClick={onExport}>Export</DropdownMenuItem>
      <DropdownMenuItem className="text-destructive" onClick={onDelete}>Delete</DropdownMenuItem>
    </DropdownMenuContent>
  </DropdownMenu>
</ActionBar>
```

## Performance rules

Dense profiles often have a lot of data. Don't let that make them slow.

1. **Stream the page.** Hero loads instantly with cached canonical fields. Tabs lazy-load on click.
2. **Don't fetch what you don't show.** A 50KB enrichment JSON doesn't need to come over the wire if the user only opens Activity.
3. **Virtual-scroll long timelines.** If the activity feed is 500+ events, virtualize it (TanStack Virtual / react-virtuoso).
4. **Optimistic mutations.** Click "Mark hot" → UI updates instantly → request goes in background → revert on error.
5. **Cache the AI summary** on the row, not regenerated on every view.

## What to test before shipping a dense view

- Can a new user understand what they're looking at in 5 seconds without a tour?
- Can they trigger the most common action with one click?
- Can they go from "this lead detail" to "next lead" with a keyboard?
- Does the page load (hero visible) in under 500ms?
- Does it work at 1280×800 (smallest common laptop) without horizontal scroll?
- Is anything below the fold that should be above?
- Are there fields shown that nobody uses? (delete them)
- Are there fields used that nobody can find? (move up)

## Anti-patterns to fight

1. **The "everything visible" wall.** A 1500px-tall page of fields with no hierarchy. Users will scroll once, get tired, never return.
2. **Tabs of tabs.** If you need nested tabs, your information architecture is broken.
3. **Modals that hide context.** Editing a field shouldn't take you away from the lead context.
4. **Required tour to understand the page.** If you need a coach mark, the design has a problem.
5. **"Save" buttons for fields.** Inline edit, autosave on blur, undo if needed.
6. **Free-text status fields.** Status is enum, not string. Free text leads to "qualfied", "qualified ", "Qualified" all in the database.
7. **Different layouts for similar entities.** Lead, contact, account should follow the same 4-layer template so users don't relearn.
8. **One column per source.** Don't show "Apollo Email" and "Hunter Email" as separate columns. Show the canonical email and let provenance live in the hover.
